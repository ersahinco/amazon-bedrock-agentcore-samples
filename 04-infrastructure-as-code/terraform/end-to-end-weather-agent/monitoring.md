# Observability Architecture

This document describes the 4-destination log architecture for the AgentCore weather agent,
the Terraform resources that manage each destination, and the boundary between the Python
logging layer and the OTel auto-instrumentation layer.

---

## Log Destinations

The AgentCore runtime routes telemetry to four distinct destinations. Each serves a
different purpose and has a different creation mechanism.

### 1. Vended Logs Group

| Property | Value |
|---|---|
| Log group path | `/aws/vendedlogs/bedrock-agentcore/<runtime-id>` |
| Terraform resource | `aws_cloudwatch_log_group.agent_runtime_logs` (`observability.tf`) |
| Creation mechanism | Terraform-managed |
| Content | Platform-level invocation records (`BedrockAgentCoreRuntime_ApplicationLogs`) |
| Retention | 14 days |
| Delivery pipeline | Yes — `aws_cloudwatch_log_delivery.logs` routes `APPLICATION_LOGS` from the runtime ARN to this group |

This group does **not** contain the application's `logger.info()` output. It holds
AWS platform records that describe invocation lifecycle events (start, end, errors)
generated by the AgentCore runtime itself.

### 2. App Logs Group

| Property | Value |
|---|---|
| Log group path | `/aws/bedrock-agentcore/runtimes/<runtime-id>-DEFAULT` |
| Terraform resource | `aws_cloudwatch_log_group.app_runtime_logs` (`observability.tf`) |
| Creation mechanism | Auto-created by the AgentCore runtime; Terraform adopts the existing group to enforce retention |
| Content | Structured JSON from `StructuredFormatter` + OTel span records (stream: `otel-rt-logs`) |
| Retention | 14 days (Terraform-enforced, matching Vended Logs Group) |

This is where `logger.info()`, `logger.error()`, and other application-level log calls
land. Each entry is a JSON object produced by `StructuredFormatter` in `weather_agent.py`
with fields: `timestamp`, `level`, `message`, `logger`, and optional context fields
(`event`, `city`, `tool_name`, `requestId`, `sessionId`, etc.).

### 3. OTel Traces (X-Ray)

| Property | Value |
|---|---|
| Destination | `aws/spans` (X-Ray) |
| Terraform resource | `aws_cloudwatch_log_delivery.traces` (`observability.tf`) |
| Creation mechanism | Terraform-managed delivery pipeline with `TRACES` log type |
| Content | Distributed traces from `amazon-opentelemetry-distro` auto-instrumentation |
| Captures | Every Bedrock API call, tool execution, browser session, and code interpreter call with timing and token counts |

Known noise sources:
- `telemetry.extended` attribute on every span increases payload size.
- Parallel `browser_use` retries produce duplicate `chat anthropic.claude-3-5-haiku-*`
  Converse spans (up to 4x per logical call).

### 4. Application Signals

| Property | Value |
|---|---|
| Destination | `aws/application-signals` (CloudWatch) |
| Creation mechanism | AWS-managed (derived from OTel spans automatically) |
| Content | Pre-aggregated EMF metrics (namespace: `ApplicationSignals`) |
| Tracks | Latency, errors, throttles, and token counts per operation |

Application Signals are **not** raw logs. They are metrics derived from the OTel span
stream. No Terraform resources manage this destination directly — it is a downstream
effect of the traces delivery pipeline.

## OTel Layer Boundary

A critical architectural detail: the Python logging layer and the OTel instrumentation
layer are independent.

```
┌─────────────────────────────────────────────────────┐
│  Python logging layer                               │
│  weather_agent.py → StructuredFormatter → stdout     │
│                                                     │
│  Logger suppression list applies HERE:              │
│    botocore, boto3, urllib3, langchain_aws, httpx,  │
│    httpcore, posthog, opentelemetry,                │
│    browser_use.sync.service                         │
│  (all set to ERROR level)                           │
└─────────────────────────────────────────────────────┘
                        │
                        │  stdout / stderr
                        ▼
┌─────────────────────────────────────────────────────┐
│  OTel auto-instrumentation layer                    │
│  amazon-opentelemetry-distro (ADOT)                 │
│                                                     │
│  Runs BELOW the Python logging layer.               │
│  Patches botocore at the HTTP client level.         │
│  Python logger suppression has NO effect on spans.  │
│                                                     │
│  Span noise must be addressed via OTEL_* env vars   │
│  in main.tf, or via a custom SpanProcessor/Sampler. │
└─────────────────────────────────────────────────────┘
```

Setting `logging.getLogger("botocore").setLevel(logging.ERROR)` in `weather_agent.py`
silences botocore's Python log output. It does **not** affect the OTel botocore
instrumentation, which operates by monkey-patching the botocore HTTP client — a layer
below Python's `logging` module. To reduce OTel span noise, configuration must happen
at the OTel level (environment variables, custom processors, or sampler configuration).

## IAM Coverage

The existing IAM policy in `iam.tf` grants `logs:CreateLogGroup`, `logs:CreateLogStream`,
`logs:PutLogEvents`, and `logs:DescribeLogGroups` on:

```
arn:aws:logs:<region>:<account>:log-group:/aws/bedrock-agentcore/runtimes/*
```

This covers both the auto-created `App_Logs_Group` and the delivery pipeline extension.
No additional IAM changes were needed for the observability additions.

## Confidence Levels

| Component | Confidence | Notes |
|---|---|---|
| Vended Logs Group + delivery pipeline | High | Fully documented AWS pattern |
| App Logs Group retention (Terraform adopt) | High | Standard `aws_cloudwatch_log_group` behavior |
| App Logs delivery pipeline | Medium | Plausible extrapolation; not fully documented for runtime resources via SDK |
| Traces delivery pipeline (X-Ray) | High | Fully documented AWS pattern |
| Application Signals | High | AWS-managed, no Terraform needed |
| `OTEL_PYTHON_DISABLED_INSTRUMENTATIONS=botocore` | Low | Disables all botocore instrumentation including legitimate traces; exploratory |
| `OTEL_RESOURCE_ATTRIBUTES` | Low | ADOT distro may override; effect on `telemetry.extended` unconfirmed |
